# 2026-07-12 project review and phased remediation plan

## Purpose

This document converts the 2026-07-12 repository review into an ordered,
implementation-ready remediation plan. It fixes the decisions needed to resolve
the concerns so implementation agents do not have to choose product behavior,
compatibility policy, security posture, persistence semantics, or test scope.

The requirements, accepted ADRs, and policies in `AGENTS.md` remain the source
of truth. If an implementation detail below conflicts with a requirement or an
accepted ADR, stop that phase, correct this review and the affected ADR first,
and then resume. Do not silently choose a third behavior.

This is not an MVP plan. Every required capability remains part of the release
contract. Progress is measured only by completed behavior and acceptance
criteria; no calendar estimates belong in this document or derived work.

## Status vocabulary

Only these phase states are used:

- `TODO`: any implementation, test, documentation, ADR, or quality-gate item in
  the phase is incomplete. Work in progress remains `TODO`.
- `COMPLETE`: every task and every phase-specific and global definition-of-done
  item has passed, and the evidence links are in the phase map.

Do not mark a phase `COMPLETE` based on code alone. Do not add partial,
in-progress, blocked, deferred, or not-applicable states to the phase map.

## Concern inventory

### C-01: untrusted metainfo can consume unbounded stack, memory, and disk-read resources

`crates/swarmotter-core/src/bencode.rs` accepts recursive input without a depth
or node budget, performs unchecked string-bound arithmetic, and does not require
the decoder to consume the entire input. `crates/swarmotter-core/src/meta.rs`
does not impose a complete metainfo shape budget. Engine paths narrow the
declared piece length and allocate piece buffers after parsing. Watch-folder
imports read an entire file before applying the API/BEP 9 metadata-size policy.
Malformed durable piece-hash data can reach an exact-slice copy without first
checking the decoded length.

### C-02: the default containment posture and live failure behavior do not meet the fail-closed claim

`NetworkConfig::default()` selects disabled containment while Serde's partial
`[network]` default selects strict mode. Starting without a configuration file
therefore enables the torrent data plane without an explicit operator choice,
and getting-started documentation recommends disabled mode. The live health
loop constructs its own OS probe, is not deterministically testable, and uses
normal shutdown paths after a required interface disappears. Existing UDP
sockets do not re-check a shared containment gate on send. Public statuses
`socket_bind_failed` and `blocked_fail_closed` are not reached by production
status transitions.

### C-03: compatibility APIs bypass the native API's browser-origin defense

The `/transmission/rpc` and `/api/v2` qBittorrent-compatible routes are mounted
outside the middleware that validates `Origin`, `Host`, and Fetch Metadata for
`/api/v1`. When authentication is intentionally disabled, form-compatible
mutation endpoints can accept cross-site browser requests. This diverges from
ADR-0049's same-origin protection requirement.

### C-04: per-torrent seeding behavior exists as pure logic but is not a product capability

The core has `TorrentSeeding`, but `Torrent` does not persist it and the daemon
always evaluates `TorrentSeeding::default()`. No native endpoint or Web UI
control edits it. Production completion starts a seeder without transitioning
the torrent to `seeding`, so `active_seeds` can remain zero. A seeder receives a
new upload limiter at start, while a live per-torrent limit update changes only
the downloader's limiter. Required ratio, idle, seed-forever, stop-target, live
shaping, and truthful lifecycle behavior are therefore not complete.

### C-05: `bandwidth.max_peers` is divided into estimates, not enforced globally

`effective_peer_worker_limit` rounds a global value into at least one worker per
active torrent. More active torrents than the configured cap can exceed it.
Inbound seeding sessions use a separate allowance, so inbound and outbound
sessions are not counted against one process-wide budget even though the
configuration documentation describes a total connection cap.

### C-06: watch-folder import is immediate, repeatable, blocking, and non-transactional

The pure stable-size helper is not used by the daemon. A discovered file is read
immediately with blocking filesystem calls in an async scan. `leave` files are
reprocessed every scan, including duplicate failures. Registry and queue state
change and scheduling can begin before persistence succeeds, with no rollback.
Post-import moves ignore errors and can overwrite or strand operator files
depending on platform behavior. Import history is unbounded.

### C-07: the contained HTTP reader is not a complete HTTP/1.1 implementation

Tracker and webseed traffic obtains sockets through containment, but the custom
reader treats EOF as message framing and does not decode chunked bodies, honor
`Content-Length`, implement bounded redirects, or construct every authority
correctly for explicit ports and IPv6 literals. Tracker code treats redirects
as response bodies. Webseed range verification cannot work reliably with
chunked transfer coding.

### C-08: oversized modules and duplicate declarations increase regression risk

`crates/swarmotterd/src/daemon.rs`, `crates/swarmotterd/src/engine.rs`,
`crates/swarmotter-api/src/handlers/torrents.rs`, and
`crates/swarmotter-web/assets/app.js` combine unrelated responsibilities.
`swarmotterd/src/main.rs` redeclares modules already exposed by `lib.rs`, so
daemon source and unit tests compile through both binary and library targets.

### C-09: release-completion evidence is ahead of reachable production behavior

The v1 completion tracker marks behavior complete where the inspected runtime
does not expose it end to end. Besides C-02 through C-07, examples include
tracker scrape without a scrape request implementation, per-file
`bytes_completed` remaining at initialization values, and lifecycle/status
variants that are modeled but not produced. A passing pure-helper test is not
evidence that its product path is complete.

## Phase 1: bound and harden untrusted metadata parsing

**Status: COMPLETE**

**Decision record:** create
`design/adr/0050-bounded-untrusted-metainfo-parsing.md`. Record that limits
apply equally to bencoded API uploads, watch folders, magnet metadata, and
direct core parser callers. Restored daemon state is a separate JSON trust
boundary: piece hashes use a bounded exact-length decoder and restored
`TorrentMeta` values pass shape validation before runtime use; do not describe
JSON state as passing through the bencode byte, depth, or node budgets.

### Required implementation

Define these public constants in `crates/swarmotter-core/src/meta.rs` and use
them instead of duplicate literals:

```rust
pub const MAX_TORRENT_METADATA_BYTES: usize = 16 * 1024 * 1024;
pub const MAX_BENCODE_DEPTH: usize = 128;
pub const MAX_BENCODE_NODES: usize = 250_000;
pub const MAX_TORRENT_FILES: usize = 100_000;
pub const MAX_TORRENT_PIECES: usize = 750_000;
pub const MAX_PIECE_LENGTH: u64 = 64 * 1024 * 1024;
```

Apply this exact parser contract in `bencode.rs`:

- Reject input larger than `MAX_TORRENT_METADATA_BYTES` before parsing.
- Count the root as depth zero. Increment on entering a list/dictionary and
  reject an entry that would exceed `MAX_BENCODE_DEPTH`.
- Count every integer, byte string, list, and dictionary as one node and reject
  the node that would exceed `MAX_BENCODE_NODES`.
- Use `checked_add` for every cursor/length calculation and verify the end is no
  greater than input length before slicing or allocating.
- Reject empty integers, leading zeroes other than `0`, negative zero, missing
  terminators, non-string dictionary keys, and duplicate dictionary keys.
  Continue accepting unsorted unique keys for interoperability; info-hash
  calculation still uses the original encoded `info` slice.
- Require EOF after exactly one top-level value. Trailing bytes are an error.
- Return `CoreError::Bencode` or `CoreError::MalformedTorrent` with a short
  reason. No malformed input may panic.

Apply these metainfo checks before constructing `TorrentMeta`:

- `piece length` is in `1..=MAX_PIECE_LENGTH`.
- The pieces string is an exact multiple of 20 and its hash count does not
  exceed `MAX_TORRENT_PIECES`.
- File count does not exceed `MAX_TORRENT_FILES`.
- Total-length and file-offset additions are checked.
- Empty torrents are invalid. A non-empty torrent provides exactly
  `ceil(total_length / piece_length)` hashes.

Use `u32::try_from(meta.piece_length)` at every engine/storage boundary that
needs `u32`; return `MalformedTorrent` instead of narrowing with `as`. Allocate
no piece-sized buffer until limits and conversions pass.

Before reading a watch file, check metadata length, allocate only the checked
length, and use a bounded read that rejects growth over the limit. Keep the same
limit for API and BEP 9 metadata even when `api.max_request_body_bytes` is
higher for other requests.

In durable-state deserialization, require each decoded SHA-1 piece hash to be
exactly 20 bytes before copying. Include torrent record and piece index in the
error without payload data or content paths.

## Phase 2: make containment fail safe by default and on live path loss

**Status: COMPLETE**

**Decision record:** create
`design/adr/0051-explicit-network-path-and-live-containment-gate.md`. It
supersedes only earlier text permitting an omitted network table to select
disabled containment. Link ADR-0005, ADR-0012, and ADR-0047 explicitly.

### Required configuration behavior

Make `NetworkConfig::default().mode` `strict`, matching the Serde default. An
omitted `[network]` table produces strict mode without a path and fails
`Config::validate()` with `invalid_config` before the control listener or any
background task starts. `--check-config` must fail the same way.

Disabled containment remains available only through this explicit setting:

```toml
[network]
mode = "disabled"
```

This mode is for development or a separately enforced boundary such as the
supplied Gluetun shared-network-namespace deployment. Never infer disabled mode
from a missing file/table, platform, bind failure, or unavailable interface.
Never auto-change strict to preferred or disabled.

Move full config validation before logging initialization and before the
`--check-config` success message. A run without `--config` fails unless env
overrides provide a valid strict path or explicit disabled mode.

### Required live containment architecture

Introduce one process-wide `ContainmentGate`, owned by `DaemonRuntime` and
shared by every binder, DHT runner, listener, engine, seeder, tracker, webseed,
and metadata task. Use atomics plus `tokio::sync::Notify`, not a new dependency.
It provides:

- `allow()`: permit traffic and advance generation only on blocked-to-allowed.
- `block(status, detail)`: deny operations, store status/detail, advance the
  generation, and notify waiters.
- `enforce()`: return `CoreError::NetworkBlocked` when denied.
- `cancelled_since(generation)`: complete when generation changes to blocked.

Every bind, connect, resolve, accept-loop iteration, UDP send, tracker request,
webseed request, and DHT send calls `enforce()`. Each top-level data-plane task
selects normal work against `cancelled_since(start_generation)` so connected
TCP/TLS streams drop on block. The control listener never uses this gate.

Store one injected `Arc<dyn InterfaceProbe + Send + Sync>` on `DaemonRuntime`.
Production injects `OsInterfaceProbe`; tests inject a mutable fake. Extract one
`pub(crate) async fn network_health_tick()` from the loop. The loop only invokes
that operation; tests call it directly without sleeping.

On healthy-to-unhealthy transition, while holding the data-plane transition
lock, perform this exact order:

1. Block the gate immediately.
2. Stop the inbound listener and shared DHT runner.
3. Abort/drop all downloader, metadata, tracker, webseed, and seeder task
   handles; do not await graceful protocol shutdown before dropping sockets.
4. Reconcile byte counters and verified progress already reported by tasks.
5. Set previously active torrents to `network_blocked`, retain prior activity
   for recovery, persist, and publish torrent plus network-status events.

On recovery, update health, allow the gate, reconstruct listener/DHT, and
requeue only work active before the block. Paused and automatically seed-stopped
torrents remain stopped.

Route bind/listen/source-bind failures through a runtime health-report channel.
Such a report blocks the gate and exposes `socket_bind_failed`. Use
`blocked_fail_closed` only when strict policy denies traffic and no more specific
interface/address/namespace/route/DNS/bind status applies. Both statuses require
production-path API tests.

## Phase 3: apply browser-origin protection to every control API

**Status: COMPLETE**

**Decision record:** update ADR-0044 and ADR-0049. No new ADR is needed because
this phase restores their stated behavior.

### Required implementation

Extract current `Origin`/`Host`/Fetch Metadata validation into one
`browser_origin_guard` Axum middleware. Build the router as:

- Public health routes that neither mutate nor reveal torrent data.
- All `/api/v1`, `/transmission/rpc`, and `/api/v2` routes under the shared
  origin guard, followed by each surface's existing auth/session middleware.

Apply the guard before handler extraction and before compatibility-enabled
checks. Remove the duplicate guard call from native auth.

The exact policy is:

- Reject `Sec-Fetch-Site: cross-site` and `Sec-Fetch-Site: same-site` with HTTP
  403; only `same-origin`, `none`, or an absent value can continue.
- When an ordinary browser `Origin` exists, parse it and require its authority
  to match `Host` using the existing normalized host/port comparison.
  Intentionally ignore scheme so ADR-0044's TLS-terminating reverse-proxy
  behavior remains valid. Reject malformed, opaque, `null`, multi-value, or
  mismatched origins with 403, even if the request supplies a valid API token.
- Treat one `chrome-extension://<extension-id>` Origin as a separate,
  deliberately cross-origin client class. The ID must contain exactly 32
  lowercase characters from `a` through `p`; a port, user information, path,
  query, fragment, whitespace, list, duplicate field, or invalid byte rejects
  with 403. Require one valid Host authority but do not compare it to the
  extension ID. Permit the request only when `api.require_auth = true` and
  exactly one non-duplicated credential mechanism — `Authorization: Bearer` or
  `X-SwarmOtter-Auth` — constant-time matches the current `api.auth_token`.
  Reject auth-disabled mode even when a token remains configured, and reject
  requests that send both credential mechanisms. There is no extension-ID
  allowlist or unauthenticated extension exception.
- Accept Chrome's privileged service-worker shape with `Sec-Fetch-Site: none`;
  the ordinary Fetch Metadata allowlist remains authoritative, so `same-site`
  and `cross-site` reject before the extension credential lookup.
- When both browser headers are absent, continue as a non-browser client to
  normal authentication.
- Do not weaken the rule with CORS. Authentication mode changes ordinary
  credential checks only; it affects origin handling solely for the explicit
  authenticated Chrome-extension client class above.

Keep native JSON error envelopes and each compatibility surface's documented
error shape/status. Extension credential rejection uses HTTP 403: native routes
return `extension_origin_forbidden` with instructions to enable auth and send a
valid token, Transmission returns its JSON error object with that guidance, and
qBittorrent retains plain-text `Forbidden`. Never redirect a rejected API
request to the Web UI.

### Required tests

Create one table-driven real-router matrix for authentication enabled/disabled:

- Native single add, bulk add, pause, remove, settings mutation, WebSocket, and
  SSE.
- Transmission session negotiation and a mutating RPC method.
- qBittorrent login, add, pause/resume, and delete form endpoints.
- Same origin; rejected same-site and cross-site Fetch Metadata; foreign or
  malformed origin; `Origin: null`; and absent browser headers.

Cross-site cases return 403 before mutation and leave fake-daemon counters zero.
Same-origin and headerless authenticated clients retain existing behavior.

Add a second real-router matrix for the Chrome Manifest V3 service-worker
shape. With auth enabled, the valid extension ID, `Sec-Fetch-Site: none`, valid
Host, and valid Bearer token must reach every named native and compatibility
route; `X-SwarmOtter-Auth` must separately reach native bulk add. The bulk case
must prove the fake daemon receives `add_magnet`. For every route, auth-disabled
mode, missing token, invalid token, duplicate Authorization, and both credential
mechanisms return the surface-specific 403 and leave an exact state snapshot
unchanged. Short IDs, characters outside `a`-`p`, extension ports, malformed or
duplicate Hosts, ordinary foreign HTTP(S), `null`, opaque, malformed, and
multi-value Origins remain in the rejection matrix.

## Phase 5: enforce one process-wide peer connection budget

**Status: COMPLETE**

**Decision record:** create
`design/adr/0053-process-wide-peer-session-permit-pool.md` and link ADR-0046
and ADR-0047.

### Required implementation

Replace global-share arithmetic with one runtime-owned `PeerPermitPool` backed
by `tokio::sync::Semaphore`:

- `bandwidth.max_peers == 0` means unlimited and returns an uncounted permit.
- A nonzero value creates exactly that many owned permits.
- Acquire before initiating outbound peer TCP/uTP and hold until the peer
  session drops.
- An accepted inbound peer uses `try_acquire_owned`; without a permit, close it
  before handshake and increment a denied counter.
- Metadata-fetch and all serial, parallel, and endgame peer sessions consume
  permits. Trackers, webseeds, DHT nodes, and DNS do not because this setting is
  specifically the peer connection limit.
- `max_peers_per_torrent` remains separate. A session satisfies both caps.

Delete `effective_peer_worker_limit` global division. Engine worker count may
still use the per-torrent cap; workers wait for a global permit before opening a
socket. Never hold a permit while waiting for discovery or retry.

Treat live `max_peers` changes as ADR-0047 data-plane reconstruction: block new
starts, stop peer-bearing tasks, construct a new pool, restart eligible work,
then commit settings. Do not resize a semaphore in place. Failed reconstruction
restores the old config and old-size pool.

Expose `peer_limit`, `peer_permits_in_use`, `peer_permits_available`, and
`peer_sessions_denied` diagnostics. In unlimited mode, limit is `0`, available
is `null`, and in-use still reports observed sessions.

### Required tests

- Mixed inbound/outbound sessions share the cap.
- Metadata, normal, parallel/endgame, uTP, and seeding paths hold lifetime
  permits.
- Cancellation, connect/handshake error, normal EOF, and task panic release an
  owned RAII permit; do not balance counters manually.
- Unlimited mode, composition with per-torrent cap, and diagnostics.
- Successful/failed ADR-0047 reconfiguration commit/roll back and never exceed
  either applicable nonzero cap.
- A generated local swarm samples diagnostic concurrency during the run, not

## Phase 6: make watch-folder ingestion stable, idempotent, and transactional

**Status: COMPLETE**

**Decision record:** create
`design/adr/0054-watch-folder-stability-idempotence-and-import-atomicity.md`.

### Required scanner behavior

Move the complete directory walk into `tokio::task::spawn_blocking`. Use
`symlink_metadata`, skip all symbolic links, and never descend through symlinked
directories. Return sorted paths for deterministic history.

Store an in-memory observation map keyed by a lexically normalized absolute
path relative to the watch root; do not use `canonicalize`, which follows
symlinks. An observation holds length, modified timestamp, consecutive stable
scans, and last processed fingerprint. The fingerprint is `(length,
modified_timestamp)` and is only an observation key, not a security hash.

Use this state machine:

1. First sighting records metadata and does not import.
2. Changed length/timestamp resets stable scans to one and does not import.
3. Two consecutive scans with identical metadata make the file eligible.
4. A successfully processed fingerprint is skipped until metadata changes.
5. Remove observations for paths absent from the latest successful root scan.

After restart, first observe again. If an unchanged file is already registered,
the second scan treats it as successful duplicate, executes success action,
marks processed, and does not repeat. Add no durable watch ledger.

Apply Phase 1's size bound before allocation. Re-check metadata after bounded
read; if length/timestamp changed, discard bytes, update observation, and wait
for stability again.

### Required import transaction and outcomes

Refactor API and watch add paths to one daemon mutation primitive, under the
existing mutation serialization lock, in this order:

1. Parse and run storage/network preflight without mutation.
2. Snapshot prior registry/queue entries for the hash.
3. Insert torrent and queue entry in memory.
4. Persist durable state.
5. On persistence error, restore both snapshots before releasing the lock and
   return the original error.
6. Only after persistence succeeds, publish events and schedule reconciliation.

A duplicate is `success: true`, `duplicate: true`, uses the existing hash, and
executes success action. It never changes the existing torrent, queue position,
labels, path, or settings.

Only `Bencode`, `MalformedTorrent`, `InvalidInfoHash`, and `Parse` are permanent
input failures and execute failure move. Storage, I/O, persistence, containment,
and internal errors are transient: leave source, do not mark processed, and
retry after later stability.

Post-action rules are:

- Never overwrite an archive/failure destination.
- Create a configured destination directory if absent.
- If destination exists or move/delete fails, keep the primary import outcome,
  add `post_action_error`, mark processed, and do not re-import. The operator
  resolves the source manually.
- `leave` marks the fingerprint processed after success/duplicate.

Extend `ImportResult` with `post_action_error: Option<String>` and stable
`outcome`: `imported`, `duplicate`, `permanent_failure`, or
`transient_failure`. Retain only the newest 10,000 results in insertion order;
evict oldest on insertion. Do not persist this operational history.

### Required tests

- Multi-scan partial copy and a metadata change during read.
- `leave` imports once; changed replacement processes once again.
- Duplicate after restart runs action once without changing existing torrent.
- Injected persistence failure rolls registry/queue back and publishes/schedules
  nothing.
- Permanent errors move; transient errors stay/retry.
- Destination collision never overwrites and reports `post_action_error`.
- Sorted recursive/non-recursive scans, ignored symlinks, and observation
  removal for disappeared paths.
- History entry 10,001 evicts entry 1.
- API add uses the same rollback primitive and retains response contract.

### Documentation updates

- Specify stability, restart, duplicate, error classification, non-overwrite,
  and history behavior in design/operator configuration and Web UI docs.
- Add failure injection and partial-copy cases to `design/testing.md`.
- Document the shared durable add transaction in `design/architecture.md`.
- Keep watch rows in `design/v1-completion-tracker.md` incomplete until tests
  pass.
- Record stable/idempotent import in `CHANGELOG.md`.

### Phase 6 definition of done

No unstable/oversized file is parsed; unchanged files have one terminal result
per run; duplicates are harmless; persistence leaves no ghost state; post

## Phase 7: make contained HTTP framing, redirects, ranges, and tracker scrape complete

### Required dependency and transport decision

Keep connection creation in the containment binder. Use Hyper only as an

  needed, and wrap only that stream in Hyper's Tokio I/O adapter.
- Send HTTP/1.1 origin-form and a `Host` authority retaining non-default port
  and brackets around IPv6 literals.

authorization or cookies. Preserve `Range` on webseed GET redirects.

without valid permitted `Location`, 4xx, or 5xx is a tracker failure with status
context.

Webseed accepts only final 206. Parse `Content-Range`; require start/end exactly
equal the requested inclusive range and body length equal that range. Bound
decoded bytes to expected length and fail on the first excess byte. Do not
accept 200 for a range request.

### Required tracker scrape behavior

a scrape URL only when the final path component begins with `announce`; replace
that prefix with `scrape` while preserving suffix (`announce.php` becomes
`scrape.php`). Other shapes are `unsupported`, not failed.

Send one binary percent-encoded `info_hash` parameter per requested torrent.
Parse BEP 48 `files` through Phase 1's bounded decoder. Update a tracker's
complete, incomplete, and downloaded counts only for an exact matching 20-byte
key. Missing/malformed matching data is failure and does not overwrite the last
successful values. Expose last scrape status/time/counts in tracker API/detail
and the Web UI table. Do not claim UDP scrape support without separate
requirements and acceptance tests.

# ADR-0028: mdBook Mermaid and GitHub Pages Publishing

## Status

Accepted

## Context

SwarmOtter has user-facing documentation under `docs/` and maintainer/design
documentation under `design/`. The user guide needs diagrams for deployment
and network containment clarity, and the project needs a repeatable way to
publish the mdBook output after changes merge to `main`.

Plain text diagrams are easy to maintain but do not render as diagrams in the
published guide. Publishing directly from the repository's `/docs` directory
would expose the raw Markdown source instead of the built mdBook site and would
not run the Mermaid preprocessor.

## Decision

Use mdBook as the user-guide site generator with `docs/` as the source root.
Use `mdbook-mermaid` as the mdBook preprocessor for Mermaid diagrams in
user-facing documentation.

Pin documentation build tooling in CI:

- `mdbook` version `0.5.0`.
- `mdbook-mermaid` version `0.17.0`.

Commit the Mermaid browser runtime assets generated by `mdbook-mermaid install
.` under `assets/`:

- `assets/mermaid.min.js`.
- `assets/mermaid-init.js`.

Publish the generated `book/` directory through GitHub Pages using the
repository's `CI` workflow:

- Build the book for pull requests and pushes.
- Upload the Pages artifact only for successful pushes to `main`.
- Deploy with `actions/deploy-pages` to the `github-pages` environment.
- Configure the GitHub repository Pages source to **GitHub Actions** instead
  of branch or folder publishing.

## Consequences

User-facing docs can contain real Mermaid diagrams while remaining plain
Markdown source in the repository.

The published site is generated from `docs/`; `design/` remains unpublished
maintainer and design context.

CI now depends on mdBook documentation tooling for the docs build. The tools
are build-time dependencies only and do not affect SwarmOtter runtime behavior
or torrent network containment.

The committed Mermaid assets must be refreshed when `mdbook-mermaid` is
upgraded. The generated `book/` output remains ignored and should not be
committed.

Repository administrators must set GitHub Pages to use GitHub Actions before
the first Pages deployment is served.
